# W1.2 — Pre-rank scoring at intake

> **Goal:** When a memory is captured via `musubi_remember`, run a quick
> LLM pass *before* the row is written so it lands with structured
> metadata — calibrated importance, extracted topics, callback-worthy
> flag, emotional valence. Lets fresh provisional rows surface intelligently
> on the next call without waiting for the maturation cron, and lays the
> data substrate for the EQ layer (PL.3).

## Why now

We band-aided the cross-channel fresh-save problem in v1.2.0 with
`state_filter=[provisional, matured, promoted]` — phone-Nyla can now see
something Openclaw-Nyla just saved, immediately. But that returns *every*
provisional row equally weighted. There's no signal that *this provisional
row is the cocoa-pods prank Eric just told me to remember* vs. ambient
operational noise. Pre-rank scoring is the architecturally correct fix:
attach the importance signal at write time, and let retrieve use it.

Two payoffs:

1. **Immediate UX:** retrieve can prefer high-importance / callback-worthy
   provisional rows over ordinary ones, even before maturation runs.
2. **Foundation for PL.3 (EQ layer):** the same scoring pass extracts
   valence and topics, which is exactly the substrate the Eric Profile
   (single curated row at `nyla/_shared/curated`) feeds on.

## The signal

For every `musubi_remember` call, the LLM scorer returns:

| Field | Type | Meaning |
|---|---|---|
| `importance` | int 1-10 | 1 trivial · 5 worth keeping · 8 actively load-bearing · 10 critical fact-of-life |
| `topics` | string[] | 1-5 concise lowercase nouns/keywords extracted from content |
| `callback_worthy` | bool | True only if Eric is likely to ask about this on the next call |
| `valence` | enum | `positive` · `neutral` · `concerned` · `frustrated` · `excited` |

These fields are determined from `content` (and optionally a small bit of
session context — the last few turns). Caller's `importance` argument
becomes a *fallback* if the scorer fails or times out.

## Where it computes

Two viable runtimes. Pick before implementation; doc this decision in
[[13-decisions/]] if it sticks.

### Option A — Gemini Flash via existing API key (recommended for v0)

- LiveKit + Openclaw both already have `GOOGLE_API_KEY` in env
  ([`base_agent.py:124`](../../tools/src/tools/base_agent.py) uses Gemini
  realtime; [`musubi_client.py:44`](../../sdk/src/sdk/musubi_client.py) uses
  Gemini embeddings)
- Use `gemini-2.5-flash-lite` with a structured-output JSON schema
- Latency: 200-400 ms typical. Fits within voice fillers ("got it, stored").
- Cost: ~$0.0001 per scoring call. 100 scores/day = $0.01/day.
- **Pros:** zero new infra, works from any agent host without routing
  changes, structured output natively supported.
- **Cons:** uses external API for what could be local; small ongoing cost.

### Option B — Local Ollama on the Musubi box

- Ollama already runs in the Musubi compose stack, model `qwen3:4b` warm
- Currently not exposed outside the docker network — adapters can't reach
  `:11434` directly from off-box.
- Two ways to expose:
  1. Wrap it: add `POST /v1/ops/score` to Musubi core, internal call to
     Ollama. Clean, abstract, but is a Musubi-server change (slice work,
     CI, release, deploy).
  2. Expose the port: open Ollama on the Musubi VLAN. Smallest code change,
     but the Ollama socket has no auth — anyone on the LAN can hit it.
- **Pros:** local, no per-call cost, control over the model.
- **Cons:** hosts that aren't on the Musubi VLAN can't reach it
  (laptop-from-coffee-shop calling can't score); cold-start latency if
  Ollama isn't already loaded.

**Recommendation:** **Option A for v0**. Ship it tonight, prove the
signal is useful, swap in Option B later if the cost/control story
matters. If we decide local is the long-term target, Option B.1 (the
`/v1/ops/score` wrapper) is the right shape — gives every adapter one
endpoint to call regardless of where it runs.

## How it stores

**Decision:** ride the existing `tags` field; no Musubi server change in v0.

The four scored fields encode as structured tags alongside whatever topics
the caller passed:

```
tags = [
  *caller_topics,                # e.g. ["dentist", "family"]
  *scored_topics,                # e.g. ["dentist", "appointment", "tuesday"]
  "@importance:8",               # structured tag: prefix `@`, key:value
  "@callback:true",              # only emit when scorer says yes
  "@valence:concerned",
  "nyla-voice",                  # existing speaker tag (unchanged)
]
```

Dedup topics across caller + scorer (a topic appearing in both is one tag).
The `@key:value` prefix distinguishes structured tags from free-form
topics so `_format_row` and other consumers can parse them out cleanly.

**Why tags, not new payload fields:** keeps Musubi server unchanged.
`EpisodicMemory.tags` is `list[str]` already — no schema migration. If
we later promote these out of tags into typed payload fields, the
upgrade path is "write to both for one release, then read from typed,
then drop tag form" — standard.

## How retrieve uses it

Three layers, each ship-able independently:

1. **v0 — client-side sort, no server change.** `musubi_search` retrieves
   `limit * 2` rows, sorts client-side: `@callback:true` first, then by
   `@importance:N` desc, then by score. Trivial `_score_search_row`
   helper in [`memory.py`](../../tools/src/tools/memory.py). Ships
   tonight as part of the same PR.
2. **v1 — server-side scoring boost (later slice).** Retrieve scoring
   gains an `importance` axis derived from the structured tag. Replaces
   the v0 client-side sort with proper ranking. Real Musubi slice,
   forbidden-paths-respecting, ADR if the formula matters.
3. **v2 — typed payload fields.** Promote `importance`, `valence`,
   `callback_worthy` out of tags into structured payload columns +
   indexes. Migration shipped as a slice, dual-write window, drop tags.

v0 alone is enough for the UX win Eric described. v1 and v2 are
mechanical follow-ups when retrieve quality starts caring.

## Scope of THIS lift (the morning project)

Voice-side first, Openclaw plugin later. Concrete file list:

### `openclaw-livekit/`

- **`tools/src/tools/scoring.py`** *(new)* — `score_memory(content, *, session_hint=None)` async helper. Calls Gemini with structured output schema. 1.0s timeout. Returns dict; empty dict on timeout/error (the caller treats absence of fields as "scorer didn't run, use defaults").
- **`tools/src/tools/memory.py`** *(modify)* — `musubi_remember` invokes `score_memory` before `capture_memory`. Merges scored fields into `topics`. Replaces hardcoded `importance=7` with scored-or-fallback. ~30 lines added.
- **`sdk/tests/test_tools_scoring.py`** *(new)* — unit tests for `score_memory`: happy path produces all fields, timeout returns empty dict, malformed JSON returns empty dict, schema-violating response returns empty dict.
- **`sdk/tests/test_tools_memory.py`** *(modify)* — existing tests stub `score_memory` to return a known dict; assert structured tags appear in the captured row.
- **`tools/src/tools/memory.py`** *(modify)* — `_format_search_row` parses `@importance:N` and surfaces it as a prefix when ≥8 (only): `[openclaw · ★8] cocoa pods prank ...`. Helps the LLM bias toward the high-importance hits in its narrative response.
- **`agents/{nyla,aoi,party}/prompts/system.md`** *(modify)* — one-paragraph note in each prompt explaining the `★N` annotations and the fact that `musubi_remember` now scores automatically (so the LLM doesn't try to set `importance` itself unless overriding).

### Out of scope for THIS lift

- Openclaw plugin mirror (PL.3-or-soon — same structure, different repo, do after voice-side proves the signal).
- Musubi server-side ranking change (v1 above — separate slice, ADR-worthy).
- Profile-as-curated-row (PL.3 — uses this scoring data but is its own architecture).
- Backfilling existing rows with scores (one-shot operator script later if we want it).

## Implementation plan

In order. Each step is independently sane; don't skip ahead.

1. **Decide A vs B.** If A (Gemini), continue. If B (Ollama), open an
   ADR for the `/v1/ops/score` endpoint shape and pause this work
   until that lands.
2. **Write `scoring.py`.** One async function, one timeout, one
   structured-output schema. No retry — just fail to empty dict.
3. **Unit-test `scoring.py`** in isolation. Mock the Gemini transport.
4. **Wire into `musubi_remember`.** Score → merge tags → existing
   `capture_memory` call. Fallback path is "scorer empty → use the
   caller's importance, no extra tags."
5. **Unit-test the wired `musubi_remember`** via the existing stubbed
   `_StubClient` pattern. Assert: structured tags present when scorer
   succeeds; absent when scorer returns `{}`.
6. **Surface high-importance in `_format_search_row`.** Tiny.
7. **Prompt notes.** One paragraph per voice agent.
8. **Run `make lint typecheck test`.** All green.
9. **Deploy via `scripts/deploy-agents.sh`.** Restart all three.
10. **Live smoke:** save a memory marked clearly callback-worthy
    ("remember to call John about the offer tomorrow morning") →
    inspect the row in Musubi → verify the structured tags landed →
    call Nyla back → confirm the row surfaces near the top of any
    related search.

## Open decisions

These need a yes/no before code, captured here so they don't get
re-litigated:

1. **Runtime: Gemini (A) or Ollama (B)?** Recommendation: A for v0.
2. **Caller `importance` vs scored `importance` — who wins on
   conflict?** Recommendation: caller wins. The agent's prompt says
   "Eric explicitly told me this matters" → respect it. Scorer's role
   is *adding* a signal when the agent didn't explicitly set one
   (which is the common case — voice agents don't set importance today).
3. **Session context to the scorer — pass it or just `content`?**
   Recommendation: just `content` for v0. Pass the last user turn as
   `session_hint` once we see scorer outputs missing context that the
   conversation provides.
4. **Tag prefix for structured tags — `@key:value` or another sigil?**
   Recommendation: `@key:value`. Avoids collision with operator scope
   strings (`*:r`), distinct from free topics, parseable.
5. **What if Gemini rate-limits / errors during a call?** Recommendation:
   silent degradation. Log warning, store row without scored tags. Eric
   never knows.

## Risks / things to watch

- **Latency creep on `musubi_remember`.** A 400 ms scoring call inside
  the user-perceived "got it, stored" window is fine. A 1500 ms call is
  not. Hard timeout at 1.0s; instrument and review after a few real saves.
- **Gemini API key exhaustion.** If Eric's Gemini quota is shared with
  voice realtime, scoring eats budget. Cheap (Flash Lite is sub-cent)
  but worth knowing. If it becomes real, switch to Option B.
- **Scorer drift over time.** Today's "importance:9" might be different
  from next month's. Calibration is unstable across model versions.
  Mitigation: the prompt is in this repo, version-controlled; a model
  bump is a deliberate change, not an accident.
- **Tag explosion.** Five scored topics + caller topics + structured
  tags = a row can have 15+ tags. Qdrant handles it but it's noise in
  raw inspection. If it bites, cap topics at 3-5.

## Success criteria (DoD-lite)

- [ ] A real `musubi_remember` call from voice produces a row in
      `nyla/voice/episodic` with `@importance:N`, `@valence:STRING`,
      and (when applicable) `@callback:true` in the tag list.
- [ ] When scoring times out or errors, the row still lands with
      caller-provided importance and no structured tags. The agent
      doesn't notice.
- [ ] `musubi_search` rows render with `★N` for high-importance hits.
- [ ] All voice agents redeployed; trace logs show scoring latency.
- [ ] Live smoke call confirms a callback-worthy save surfaces on a
      follow-up call within 5 minutes.
